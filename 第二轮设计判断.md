# 第二轮设计判断

## 分析链路

orthogonal-thinking → composition-thinking → murphy-thinking → lazyeval-thinking → spiral-thinking

## 正交分解：四个独立改进轴

7 个纠缠的改进点收敛为 4 个独立维度：

| 轴 | 名称 | 覆盖 | 优先级 |
|----|------|------|--------|
| A | 优化决策质量 | 触发准确度、超时策略 | P0 |
| B | 缓存表示广度 | 单命令→序列缓存、命中率 | P0→P2（分阶段） |
| C | 上下文利用深度 | goal/constratins 提取 | P2（推迟） |
| D | 验证严谨度 | TDD 测试案例 | P1 |

## 组件选择（Iron Sharpens Iron）

**v2.1 入选：**
- S0: 硬防护 H1-H4（时间断路器、ERROR trap、原子写入、jq 容错）
- S1: Test Corpus（5 个已知命令模式引导，不等"真实收集"）
- S3: Complexity Scorer（score_complexity() 1-5，启发式阈值 ≥3）
- S4: Copilot Latency Tracker（滑动窗口平均，>10s 自动禁用）
- S5: Simple-Cached Exception（score≤2 但 conf≥0.7 → 仍用缓存）

**推迟到 Turn 3：**
- Session History (S2)：无即时消费者，纯 holding cost
- Pair Cache (B1)：需要 session 数据积累
- Goal Extractor (C)：NLP 在 bash 中太重
- 复杂度阈值自校准：需要 scorer 先部署收集数据
- Markov 转移矩阵 (B3)：需要大量 pair 数据

## Murphy 压力测试：致命场景与硬防护

| 场景 | 严重度 | 防护 | 机制 |
|------|--------|------|------|
| Hook 超时级联（新增逻辑 > 15s → Claude 杀 hook → 所有 Bash 被阻） | **致命** | H1 时间断路器 | 每阶段检查 `$SECONDS`，> 10s 立即 `exit 0` 透传 |
| 意外错误 → exit ≠ 0（Claude 理解为"阻止调用"） | **致命** | H2 ERROR trap | `trap 'echo "$ORIGINAL_INPUT"; exit 0' ERR` |
| Bridge state 竞态损坏（并发写 → 反馈断裂） | 高 | H3 原子写入 | 所有文件写改为 `write → .tmp → mv` |
| jq 解析失败 → post-tool-use 崩溃 | 高 | H4 jq 容错 | 所有 jq 调用包装 `\|\| true` |

## LazyEval 依赖精炼

关键发现：**Session History (S2) 是最大的 premature dependency。**
- 每次 hook 调用增加 I/O 开销，但 Turn 2 中零消费者
- 延迟到 Turn 3 pair cache 实施时再添加

真实关键路径（压缩后）：
```
Time-slice 1 (并行): S0 + S1 + S3 + S4
Time-slice 2 (顺序): S5（唯一真实依赖：S3 已就位）
Gate: S6 集成验证
```

## 执行序列

1. S0 硬防护（零依赖）
2. S1 Test Corpus + S3 Complexity Scorer + S4 Latency Tracker（可并行）
3. S5 Simple-Cached Exception（依赖 S3）
4. 集成验证 + README 更新

## Turn 2 → Turn 3 积累

- Session 骨架序列数据（Turn 3 开始收集）
- Complexity score × 优化价值配对 → 阈值自校准
- Copilot 响应时间分布 → 动态延迟预算
- Test corpus 基线指标 → 回归保护
