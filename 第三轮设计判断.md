# 第三轮设计判断

## 分析链路

orthogonal-thinking → composition-thinking → murphy-thinking → lazyeval-thinking → spiral-thinking

## 正交分解：四个独立改进轴

Turn 3 候选池包含 ~10 个改进点，收敛为 4 个独立维度：

| 轴 | 名称 | 覆盖 | 优先级 |
|----|------|------|--------|
| I | Intent Signaling (Claude → Bridge) | CLAUDE.md 指令、[GOAL: ...] 标记、GOAL heredoc | P0 |
| II | Temporal Context (Bridge Memory) | Session History、Pair Cache、Markov 转移矩阵 | P1 |
| III | Prompt Construction (Bridge → Copilot) | Enhanced prompt builder、goal 注入、任务上下文 | P0 |
| IV | Self-Calibration (Bridge Meta) | 复杂度阈值自校准、置信度阈值调优 | P2 |

**独立性验证：**

- Axis I (Intent Signaling) 改变 Claude 如何表达意图，不依赖 bridge 内部记忆 (Axis II) 或 Copilot prompt 构造 (Axis III)
- Axis II (Temporal Context) 改变 bridge 记住什么，不依赖 Claude 是否使用 [GOAL:] 标记
- Axis III (Prompt Construction) 改变 Copilot 收到什么，可以从 Axis I 或 description 获取输入——有 Axis I 更好，没有也能工作
- Axis IV (Self-Calibration) 纯粹观测+调参，不改变任何功能逻辑

这四个轴是真正独立的：你可以在不改动 session history 的情况下改进 intent signaling，也可以在不改动 prompt 的情况下添加 pair cache。

### 为什么 MDP、NLP 没有成为独立轴

- **Markov 转移矩阵 (B3)**：是 Axis II 的一种实现策略，不是独立维度。Pair cache 是更简单、更直接的 temporal context 利用方式
- **NLP goal extraction (C)**：是 Axis I 的一种实现策略，不是独立维度。[GOAL: ...] 标记是更轻量、更可靠的 intent signaling 方式
- **GOAL heredoc**：是 Axis I 的一种实现策略，是对 [GOAL: ...] 标记的增强，不是独立维度

**关键洞察**：Turn 2 推迟了 5 个组件，Turn 3 又新增了 3 个 intent protocol 组件。如果不做正交分解，看起来像 8 个独立任务。分解后只有 4 个决策——而其中 2 个（Axis III、Axis IV）的大部分工作已被 Turn 2 的 copilot.sh 和 common.sh 铺垫好。

## 组件选择（Iron Sharpens Iron）

### 设计意图

**"This composition aims to enforce specialization between Claude and Copilot by creating a protocol boundary where Claude signals intent and Copilot executes optimally, so that neither system wastes capability compensating for the other's absence."**

- **Positive feedback (pull)**: Claude sees [GOAL:] markers work → invests less cognitive budget in bash syntax → produces richer intent descriptions → Copilot gets better context → produces better optimizations → Claude trusts the bridge more → uses [GOAL:] more
- **Negative feedback (push)**: Bridge detects no [GOAL:] marker → falls back to description-only prompt → optimization quality drops → Claude's unoptimized commands remain slow → incentive to adopt [GOAL:]

### v3.0 入选（P0 — Intent Protocol）

**I1: CLAUDE.md instruction**
- 告诉 Claude bridge 的存在、[GOAL: ...] 标记的用法、何时使用
- 零实现成本，最高杠杆——闭合 coordination trap
- 措辞保守："when your bash command has a non-obvious goal, prepend `[GOAL: ...]` to your description"
- 不要求 Claude "不要写 bash"——只要求"在 description 中附加意图"

**I2: [GOAL: ...] marker parsing**
- 在 pre-tool-use.sh 中解析 description 中的 `[GOAL: ...]` 标记
- 提取 goal 文本，与命令本身分离
- 轻量 regex，向后兼容——没有 [GOAL:] 时完全不影响现有行为

**P1: Enhanced prompt builder**
- 修改 copilot.sh 的 `optimize_command()` 中的 prompt 构造
- 当有 goal 时：`"Task goal: ${goal}. Optimize this shell command for the goal: ${cmd}. Output ONLY the optimized command."`
- 当无 goal 时：保持现有 prompt 不变
- goal 截断到 200 字符，防止 prompt 过长

**三者形成闭环：**
```
CLAUDE.md ──► Claude 学会 [GOAL: ...] ──► description 中出现 [GOAL: find all TODO markers]
                                                    │
                                                    ▼
                              pre-tool-use.sh 解析 [GOAL: ...] ──► goal = "find all TODO markers"
                                                    │
                                                    ▼
                              copilot.sh 构造增强 prompt ──► Copilot 收到完整意图
                                                    │
                                                    ▼
                              更精准的优化 ──► Claude 看到效果 ──► 更积极使用 [GOAL:]
```

### v3.1 入选（P1 — Session Awareness）

**S2: Session History**
- 跟踪最近 N=5 个 (skeleton, goal, timestamp) 三元组
- 存储为 `${HOOK_DIR}/session.jsonl`，每次 hook 调用 append 一行
- 边界：最多保留 5 条，超过时删除最旧的
- 只记录 score ≥ 3 的命令（简单命令不带信息量）

**B1: Pair Cache**
- 在 variants.jsonl 中增加 pair 维度
- Key: `(prev_skeleton, current_skeleton)` → cached optimized_command
- 查询逻辑：先查 pair cache，miss 则查 single cache，再 miss 则 Copilot CLI
- Pair cache 置信度独立维护，初始 0.5，反馈机制与 single cache 相同
- 边界：最多 50 条 pair 记录，超过时删除最旧的

**为什么 pair cache 优先于 Markov：**
- Pair cache 是确定性的：相同的 (prev, curr) 对 → 相同的优化
- Markov 是概率性的：需要大量数据才能建立可靠的转移概率
- Pair cache 可以立即从 session history 中学习
- Markov 推迟到 Turn 4+，当 variants.jsonl 积累足够数据后

### 推迟到 Turn 4+

| 组件 | 推迟理由 |
|------|---------|
| GOAL heredoc (`: <<'GOAL'`) | [GOAL: ...] 标记满足当前需求；heredoc 增加解析复杂度且向后兼容性存疑 |
| Markov 转移矩阵 (B3) | 需要大量 pair 数据；pair cache 先用起来 |
| NLP goal extraction (C) | Bash 中实现过重；[GOAL: ...] 标记是更可靠的信号 |
| 复杂度阈值自校准 | 需要 scorer 先部署收集数据；当前固定阈值 ≤2 经 44 测试验证合理 |
| 置信度阈值自校准 | 当前 0.4/0.7 阈值基于合理推断；需要更多 feedback.jsonl 数据 |

## Murphy 压力测试：致命场景与防护

| 场景 | 严重度 | 防护 | 机制 |
|------|--------|------|------|
| CLAUDE.md 指令使 Claude 产生幻觉（写 `[GOAL: ...]` 但不写可执行命令） | **致命** | 指令措辞保守 | 明确 Claude 仍须写可执行 bash；[GOAL:] 只是附加上下文，不是替代 |
| [GOAL: ...] 解析误匹配（description 中含 `[GOAL:` 但不是标记） | 中 | 严格 regex | `^\[GOAL:\s*([^]]+)\]` — 只匹配行首或显式前缀 |
| Goal 注入使 Copilot prompt 过长 → 超时 | 中 | 截断 | goal 截断到 200 字符；总 prompt 不超过 500 字符 |
| Session history 文件 I/O 增加延迟 | 中 | 小边界 + 原子追加 | 5 条上限，每次只 append 一行 JSON；不使用 jq 读写 |
| Pair cache 爆炸（session 产生 50+ 不同 pair） | 中 | 硬上限 + LRU | 50 条上限；超过时删除最旧的 10 条 |
| Session history 跨 session 污染（上一个 session 的 skeletons 存在文件中） | 低 | Session ID | 每次 Claude Code 启动时写 session_id，旧 session 数据在读取时被忽略 |
| Bridge state 仍可能竞态损坏 | 低 | H3 已防护 | Turn 2 的原子写入继续保护 |
| 时间预算超过 (H1) | 低 | H1+H2 已防护 | Turn 2 的硬防护继续保护 |

### 新增致命场景分析

**场景 1：Claude 写出 `[GOAL: find todos] grep` → bridge 解析 goal → Copilot 基于 goal 优化 → 命令语义改变**

这是 Turn 3 引入的新风险类别：**语义漂移**。原始命令和 optimized 命令可能因为 goal 注入而产生不同的语义。

**防护**：
- Copilot 的 prompt 中包含 `"${cmd}"` 作为必须保持语义的基础——goal 是 context，cmd 是 ground truth
- Prompt 措辞：`"optimize this shell command: ${cmd}"` — Copilot 的任务是优化命令，不是重新发明它
- 反馈循环（Turn 2 已有）：如果 optimized 命令 exit_code ≠ 0，confidence -0.2，下次可能回退
- PostToolUse 中对比原始命令和优化后命令的输出——如果差异显著，降低 confidence

**场景 2：Session history 中的 skeleton 泄露用户敏感信息（路径、文件名）**

Skeleton 已通过 `extract_skeleton()` 归一化——具体路径替换为 `<path>`，字符串替换为 `<literal>`。泄露风险低。

### 回退不变量

Turn 3 的所有新增逻辑遵循与 Turn 2 相同的回退原则：

- [GOAL: ...] 解析失败 → 使用原始 description
- Session history 读取失败 → 当作无历史
- Pair cache 查询失败 → 回退到 single cache
- Enhanced prompt 构造失败 → 使用原始 prompt
- 任何异常 → `exit 0` + 原样输出 (H2)

## LazyEval 依赖精炼

### 真实关键路径

Turn 3 的依赖图比 Turn 2 更简单——大多数组件可以并行：

```
Phase 1 (P0 — 并行):
  I1: CLAUDE.md instruction      ← 零依赖，纯文档
  I2: [GOAL: ...] parsing        ← 零依赖，纯 regex
  P1: Enhanced prompt builder     ← 形式依赖 I2（无 goal 时 fallback）

Phase 2 (P1 — 顺序):
  S2: Session history             ← 零依赖（独立文件）
  B1: Pair cache                  ← 依赖 S2（需要 session 数据查询）

Phase 3 (验证):
  测试 + sandbox 更新             ← 依赖 Phase 1 + Phase 2
  README 更新                     ← 依赖全部
```

### 哪些是真正的依赖，哪些是假设

| 依赖 | 类型 | 判断 |
|------|------|------|
| P1 依赖 I2 | **假性依赖** | P1 在无 goal 时使用 description 原文，可以独立开发测试 |
| B1 依赖 S2 | **真依赖** | B1 查询需要 session context，但可以先做 API 设计（接口 mock）再并行实现 |
| S2 依赖 variants.jsonl | **假性依赖** | S2 维护独立 session.jsonl，不读写 variants.jsonl |
| I2 依赖 pre-tool-use.sh 重构 | **形式依赖** | I2 加 10 行解析逻辑，不需要重构现有流程 |

### 关键发现：Phase 1 可以无摩擦并行

I1、I2、P1 三个组件可以同时开发：
- I1 是纯文本（CLAUDE.md），不需要改任何代码
- I2 是 pre-tool-use.sh 中加一个 goal 提取函数
- P1 是 copilot.sh 中改 prompt 模板

这是 Turn 2 lazyeval 决策的直接红利：没有在 Turn 2 添加 Session History，所以 Turn 3 的 Phase 1 完全没有时序依赖。

## 附录：关键设计权衡

### [GOAL: ...] vs GOAL heredoc

| 维度 | [GOAL: ...] 标记 | GOAL heredoc |
|------|------------------|--------------|
| 解析复杂度 | 简单 regex | 需要多行解析 |
| 向后兼容 | 完全兼容 | description 字段不支持多行 |
| Claude 采用门槛 | 低（一行文本） | 中（需要了解 heredoc 语法） |
| 信息容量 | 一句话 (~200 chars) | 可容纳多行详细描述 |
| 被误解析风险 | 中（[GOAL: 可能出现在普通文本中） | 低（heredoc 语法独特） |

**决策**：v3.0 使用 [GOAL: ...] 标记。GOAL heredoc 推迟到 Turn 4，在积累使用数据后再评估。

### Pair Cache vs Markov 转移矩阵

| 维度 | Pair Cache | Markov |
|------|-----------|--------|
| 数据需求 | 单次 (prev, curr) 出现即可缓存 | 需要大量转移数据估计概率 |
| 匹配方式 | 精确匹配 (prev_skeleton, curr_skeleton) | 概率匹配 P(curr \| prev) |
| 冷启动 | 第一次出现即记录 | 需要 N≥20 同类转移 |
| 泛化能力 | 无（只能精确匹配） | 有（概率分布允许近似匹配） |
| 实现复杂度 | 低（扩展 variants.sh） | 高（需要矩阵存储、概率计算） |

**决策**：v3.1 使用 Pair Cache。Markov 推迟到 Turn 4，当 variants.jsonl 积累足够数据后作为 Pair Cache 的泛化增强。
